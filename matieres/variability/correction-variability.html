Correction — IT1 Software Product Lines

Final Exam — January 2024

✅ THÈME 1 — Introduction

1) What is the main benefit of adopting software product lines?

Réponse correcte : A

Justification

Le bénéfice central d’une Software Product Line (SPL) est la réutilisation systématique des core assets, ce qui entraîne :

  • Réduction des coûts de développement à moyen et long terme
  • Économies d’échelle
  • Amélioration de la qualité

Les autres propositions sont incorrectes :

  • B : faux (la SPL vise à réduire les bugs)
  • C : faux (elle gère la variabilité, elle ne la réduit pas)
  • D : faux (la complexité initiale augmente, mais ce n’est pas le bénéfice principal)

2) Not a direct economic consequence

Réponse correcte : D

Justification

Conséquences économiques directes :

  • Réduction du time-to-market
  • Diminution des coûts de maintenance
  • Amélioration de la satisfaction client

L’augmentation des coûts de formation est un coût organisationnel possible, mais pas une conséquence économique directe du modèle SPL.

3) Business model statements

Réponses correctes : A, B, C, D

Justification

Le modèle économique SPL repose sur :

  • A ✓ Réduction du temps de développement des nouveaux produits
  • B ✓ Augmentation des coûts initiaux (coût plateforme)
  • C ✓ Réduction des coûts opérationnels à long terme
  • D ✓ Amélioration globale de la qualité

Tous ces éléments sont cohérents avec le modèle SPL (fort investissement initial + retour sur investissement progressif).

4) Effect on customer-supplier relationship

Réponse correcte : C

Justification

La SPL permet :

  • Une réponse plus rapide aux demandes client
  • Une configuration flexible
  • Une industrialisation des variantes

Elle n’augmente pas la dépendance ni ne diminue la qualité.

5) Cost of developing 4 systems

Données :

  • Coût indépendant = 100
  • Coût plateforme = 200
  • Réduction de 50% → coût par système = 50

Calcul :

200 + (4 × 50) = 200 + 200 = 400

Réponse correcte : A

6) Break-even point

Coût indépendant : 100n
Coût SPL : 200 + 50n

Équation :

100n = 200 + 50n
50n = 200
n = 4

Réponse correcte : B

7) Nouveau scénario

Données :

  • Coût indépendant = 120
  • Réduction 40% → coût SPL par système = 120 × 0.6 = 72
  • Coût plateforme = 360

Calcul pour 3 systèmes :

360 + (3 × 72) = 360 + 216 = 576

Réponse correcte : D

✅ THÈME 2 — Notations et Domain Engineering

1) Notation spécifique variabilité

Réponse correcte : B (FODA)

FODA est la notation historique des Feature Models utilisée pour modéliser la variabilité.

2) Dead feature

Réponse correcte : B

Une dead feature est une caractéristique qui ne peut jamais être sélectionnée dans aucune configuration valide.

3) Full mandatory feature

Réponse correcte : C

Une full mandatory feature est une feature optionnelle structurellement mais rendue obligatoire par les contraintes croisées.

4) Défauts typiques des feature models

Réponses correctes : A, B, D

  • A ✓ Dead features
  • B ✓ Full mandatory features
  • D ✓ Contraintes circulaires

Les features sans interdépendance ne constituent pas un défaut.

Une correction rigoureuse nécessite le diagramme.

✅ THÈME 3 — Application Engineering

1) Configuration

Réponse correcte : A

Configuration = sélection d’un ensemble de features pour produire un produit spécifique.

2) Derivation

Réponse correcte : B

Derivation = assemblage du produit final à partir des features sélectionnées.

3) Activités en application engineering

Réponses correctes : B, C

  • B ✓ Configuration des features
  • C ✓ Assemblage du produit

Pourquoi pas A et D ?

Question concernée :

What are the activities involved in the application engineering of software product lines?

Propositions :
A) Development of core assets
B) Configuring features for a specific product
C) Assembling the final product
D) Maintenance of existing products

Réponse donnée : B et C

1️⃣ Rappel fondamental : séparation SPL

Une Software Product Line repose sur deux processus distincts :

Domain Engineering

Objectif : créer la plateforme commune.

Activités :

  • Analyse du domaine
  • Modélisation de la variabilité
  • Développement des core assets
  • Architecture de référence
  • Implémentation des composants réutilisables

Application Engineering

Objectif : produire un produit spécifique.

Activités :

  • Configuration (sélection des features)
  • Vérification des contraintes
  • Dérivation
  • Assemblage du produit

2️⃣ Pourquoi A est faux ?

A) Development of core assets

Le développement des core assets appartient au Domain Engineering, pas à l’Application Engineering.

C’est une erreur fréquente car on associe intuitivement "développement" à toute activité SPL.
Mais dans la définition académique stricte :

Application Engineering ne crée pas les assets,
elle les réutilise.

Donc A est faux.

3️⃣ Pourquoi D est faux ?

D) Maintenance of existing products

La maintenance :

  • Peut exister dans le cycle de vie global SPL
  • Mais ne fait pas partie des activités centrales de l’Application Engineering

Dans le modèle théorique SPL :

Application Engineering = configuration + dérivation.

La maintenance relève :

  • soit du cycle de vie produit classique,
  • soit d’un processus d’évolution de la product line.

Donc D n’est pas une activité constitutive de l’application engineering.

4️⃣ Pourquoi B et C sont correctes ?

B) Configuring features for a specific product
C) Assembling the final product

Ces deux étapes correspondent exactement à :

  • Configuration
  • Derivation

Qui sont les deux piliers de l’Application Engineering.

5️⃣ Résumé conceptuel important

Beaucoup d’étudiants confondent :

"Développement dans une SPL"
avec
"Application Engineering"

Or :

Domain Engineering = produire les briques
Application Engineering = assembler les briques

C’est cette distinction qui est testée ici.

Conclusion

A est faux → développement des core assets = Domain Engineering
D est faux → maintenance ≠ activité centrale d’Application Engineering
B et C sont correctes → configuration + dérivation

4) Main purpose

Réponse correcte : D

L’objectif principal est de produire des instances spécifiques basées sur des configurations valides.

✅ THÈME 4 — Application-Requirements Matrix

Question

Comment utiliser une application-requirements matrix pour construire un modèle SPL ?

Réponse argumentée

L’application-requirements matrix permet d’identifier :

  1. Les exigences communes
  2. Les exigences variables
  3. Le degré de variabilité

Méthodologie

  1. Lister les applications (colonnes)
  2. Lister les exigences (lignes)
  3. Marquer les exigences obligatoires pour chaque application

Analyse

  • Exigence présente dans toutes les applications → Feature mandatory
  • Exigence présente dans certaines → Feature optional
  • Exigences mutuellement exclusives → Alternative group
  • Regroupements fréquents → Feature group candidate

Exemple

Applications : A1, A2, A3

Requirement A1 A2 A3
Login X X X
Payment X X
Export PDF X

Déduction :

  • Login = mandatory
  • Payment = optional
  • Export PDF = optional

Feature Model :

Root
├── Login (mandatory)
├── Payment (optional)
└── Export PDF (optional)

Conclusion

L’application-requirements matrix constitue une étape intermédiaire entre :

Application analysis → Domain engineering → Feature modeling

Elle permet d’objectiver la commonalité et la variabilité avant la construction formelle du feature model.
