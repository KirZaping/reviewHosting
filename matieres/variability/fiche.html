1. Introduction à la variabilité

Définition de la variabilité

La variabilité logicielle désigne la capacité d’un système logiciel à être configuré, adapté ou décliné afin de produire plusieurs produits différents à partir d’un socle commun. Elle permet d’exprimer explicitement ce qui varie et ce qui est commun au sein d’une famille de logiciels.

La variabilité n’est pas un accident : elle est structurelle, planifiée et maîtrisée.

Pourquoi la variabilité est nécessaire

  • Répondre à des besoins clients hétérogènes
  • Adapter un logiciel à des contextes techniques ou réglementaires
  • Réduire les coûts de développement par réutilisation
  • Industrialiser la production de logiciels similaires
  • Éviter la duplication et la divergence de code

Logiciel monolithique vs variantes vs famille de produits

Approche Description Limites
Logiciel monolithique Un produit unique, figé Peu adaptable
Variants Copies modifiées du même logiciel Explosion des versions
Famille de produits Ensemble structuré de produits Complexité à modéliser

Notion de Software Product Line (SPL)

Une Software Product Line est un ensemble de systèmes logiciels :

  • partageant un noyau commun
  • différant par des features
  • développés à partir d’actifs réutilisables
  • selon un processus industriel

2. Concepts fondamentaux de la variabilité

Feature

Une feature est une caractéristique fonctionnelle ou non fonctionnelle visible du point de vue utilisateur ou système.

Exemples :

  • Paiement par carte
  • Authentification forte
  • Mode hors ligne

Feature obligatoire / optionnelle

  • Obligatoire : présente dans tous les produits
  • Optionnelle : présente seulement dans certains produits

Variabilité positive / négative

  • Positive : ajout de fonctionnalités
  • Négative : retrait ou désactivation de fonctionnalités

Point de variation

Un point de variation est un emplacement où un choix doit être fait entre plusieurs alternatives.

Variante

Une variante est une possibilité concrète associée à un point de variation.

Configuration produit

Une configuration est un ensemble cohérent de features sélectionnées, respectant toutes les contraintes du modèle.

Dérivation de produit

La dérivation consiste à :

  1. Choisir une configuration valide
  2. Générer automatiquement le produit correspondant

Pièges fréquents à l’examen

  • Confondre feature et variante
  • Oublier les contraintes
  • Considérer une configuration partielle comme valide

3. Feature Models et FODA

3.a Définition

Un Feature Model est une représentation structurée de la variabilité d’une SPL sous forme d’un arbre de features, enrichi de contraintes logiques.

Origine de FODA

FODA (Feature-Oriented Domain Analysis) est une méthode issue de l’ingénierie des domaines visant à :

  • Identifier les fonctionnalités communes et variables
  • Structurer la variabilité
  • Servir de base à l’automatisation

Objectifs

  • Capturer la variabilité
  • Raisonner sur les configurations
  • Garantir la cohérence des produits

3.b Types de relations dans les Feature Models

Mandatory

La feature fille est toujours incluse si le parent est sélectionné.
A
└── B (mandatory)

Optional

La feature fille peut être incluse ou non.
A
└── B (optional)

Alternative (XOR)

Une seule feature doit être choisie.
A
└── (B | C | D)

Or

Une ou plusieurs features peuvent être sélectionnées.
A
└── {B, C, D}

Cardinalités de groupe

Définition plus générale :
A
└── [1..2] {B, C, D}

➡️ Au moins 1 et au plus 2 features doivent être sélectionnées.

3.c Contraintes transverses

Requires

La sélection d’une feature implique une autre.

A requires B

Excludes

Deux features ne peuvent pas coexister.

A excludes B

Rôle des contraintes

  • Capturer des dépendances non hiérarchiques
  • Réduire l’espace des configurations valides
  • Garantir la cohérence fonctionnelle

4. Construction d’un Feature Model

Méthodologie pas à pas

  1. Identifier le périmètre fonctionnel
  2. Identifier les features communes
  3. Identifier les features variables
  4. Structurer hiérarchiquement
  5. Définir les groupes (XOR, OR, cardinalités)
  6. Ajouter les contraintes transverses
  7. Valider le modèle

Bonnes pratiques

  • Features nommées de façon claire et métier
  • Hiérarchie peu profonde mais expressive
  • Contraintes explicites

Erreurs classiques

  • Arbre trop plat
  • Contraintes implicites non modélisées
  • Cardinalités incohérentes

5. Raisonnement sur les Feature Models

Configuration valide

Une configuration est valide si :

  • Toutes les contraintes sont respectées
  • Les règles hiérarchiques sont satisfaites

Configuration invalide

Une configuration viole au moins une contrainte ou relation.

Incohérences possibles

Feature morte

Feature qui ne peut apparaître dans aucune configuration valide.

Feature faussement optionnelle

Feature optionnelle en apparence mais présente dans toutes les configurations.

Modèle vide

Aucune configuration valide possible.

➡️ Ces problèmes sont critiques à l’examen.

6. Formalisation logique et SWI-Prolog

6.a Pourquoi utiliser Prolog

Prolog est un langage de programmation logique particulièrement adapté au raisonnement sur les Feature Models car il permet d’exprimer des contraintes déclaratives et de laisser le moteur logique effectuer automatiquement l’exploration de l’espace des configurations.

Ses intérêts principaux dans le cadre de la variabilité sont :

  • raisonnement logique automatique
  • vérification de cohérence des modèles
  • génération exhaustive de configurations valides
  • détection d’erreurs structurelles (features mortes, modèle vide, etc.)

6.b Traduction d’un Feature Model en Prolog

Déclaration des features 

feature(a).
feature(b).
feature(c).

Chaque feature du Feature Model est représentée par un fait feature/1.

Représentation d’une configuration

Une configuration est définie par les features sélectionnées.

selected(a).
selected(b).

Les features non sélectionnées ne sont pas déclarées comme selected/1.

Relation Mandatory

Si une feature parente est sélectionnée, la feature obligatoire doit l’être également.

Exemple : b est mandatory sous a.

invalid :-
    selected(a),
    \+ selected(b).

Contrainte Requires

La sélection d’une feature impose la sélection d’une autre.

Exemple : a requires b.

invalid :-
    selected(a),
    \+ selected(b).

Contrainte Excludes

Deux features ne peuvent pas être sélectionnées simultanément.

Exemple : a excludes c.

invalid :-
    selected(a),
    selected(c).

Groupe alternatif (XOR)

Exactement une feature doit être sélectionnée dans le groupe.

Exemple : XOR entre b et c.

invalid :-
    selected(b),
    selected(c).

invalid :-
    \+ selected(b),
    \+ selected(c).

Groupe OR

Au moins une feature doit être sélectionnée.

Exemple : OR entre b et c.

invalid :-
    \+ selected(b),
    \+ selected(c).

6.c Raisonnement et exemples en SWI-Prolog

Définition d’une configuration valide

Une configuration est valide si aucune règle invalid n’est déclenchée.

valid :-
    \+ invalid.

Vérification d’une configuration

Configuration test :

selected(a).
selected(b).

Requête :

?- valid.

Résultat attendu :

true.

Génération de configurations valides

Exploration automatique de toutes les combinaisons possibles.

choice(F) :- selected(F).
choice(F) :- feature(F), \+ selected(F).

configuration :-
    feature(F),
    choice(F),
    valid.

Détection d’une feature morte

Une feature est morte si elle n’apparaît dans aucune configuration valide.

dead_feature(F) :-
    feature(F),
    \+ ( selected(F), valid ).

Détection d’un modèle vide

Un modèle est vide s’il n’existe aucune configuration valide.

empty_model :-
    \+ valid.

Feature faussement optionnelle

Une feature est faussement optionnelle si elle apparaît dans toutes les configurations valides.

false_optional(F) :-
    feature(F),
    \+ ( valid, \+ selected(F) ).

7. Variabilité et dérivation de produits

Processus de dérivation

  1. Sélection des features
  2. Vérification logique de la configuration
  3. Association features ↔ artefacts
  4. Génération du produit final

Lien entre features et artefacts

  • code conditionnel
  • modules logiciels
  • composants
  • modèles (MDE)

Chaque feature contrôle l’inclusion ou l’exclusion d’artefacts logiciels.

Approches de dérivation

  • paramétrisation
  • composition
  • génération automatique
